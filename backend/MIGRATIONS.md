# Миграции базы данных (Alembic) — как это устроено

Этот документ объясняет, что мы настроили для автоматического применения миграций
при деплое, зачем это нужно и как с этим жить дальше.

---

## 1. В чём была проблема

В проекте одновременно есть **два способа** создавать таблицы в базе:

### Способ А — `create_all` (был изначально)

В `database.py` есть функция `create_db_and_tables()`, которая вызывает
`SQLModel.metadata.create_all(engine)`. Она запускается при старте приложения
(в `lifespan` в `main.py`).

`create_all` умеет только **одно**: создать таблицы, которых ещё нет.
Он смотрит на модели (`Task`, `Note`, ...) и создаёт недостающие таблицы.

⚠️ Чего `create_all` **НЕ умеет**:
- добавить новую колонку в уже существующую таблицу;
- изменить или удалить колонку;
- вообще как-либо менять то, что уже создано.

Именно поэтому, когда мы добавили задаче поле `title`, на проде оно бы **не появилось**:
таблица `task` там уже существует, а `create_all` в неё новую колонку не добавит.
Приложение бы упало на запросе к задачам («нет колонки title»).

### Способ Б — Alembic (миграции)

Alembic — это инструмент версионирования схемы базы. Каждое изменение схемы
(добавить колонку, создать таблицу и т.д.) описывается отдельным файлом-миграцией
в `alembic/versions/`. У каждой миграции есть:

- `revision` — её собственный id;
- `down_revision` — id предыдущей миграции (так они выстраиваются в цепочку);
- `upgrade()` — что сделать, чтобы применить изменение;
- `downgrade()` — как откатить.

Alembic хранит в базе служебную таблицу **`alembic_version`** с одной строкой —
id последней применённой миграции. Так он знает, «на какой версии» находится база,
и какие миграции ещё нужно докатить.

### Суть проблемы

Раньше деплой запускал только `create_all` и **не запускал alembic**. Для новых
таблиц этого хватало, но для изменения существующих (новая колонка `title`) — нет.
Задача: чтобы при деплое миграции применялись **автоматически**.

## 2. Что мы добавили — по файлам

### `alembic/versions/a1b2c3d4e5f6_add_task_title_and_task_tag_link.py`

Сама миграция под наши изменения. В `upgrade()` она:

- добавляет колонку `title` в таблицу `task`;
- создаёт таблицу `tasktaglink` (связь многие-ко-многим между задачами и тегами).

В `downgrade()` — обратные действия. `down_revision` указывает на предыдущую
миграцию `dac6eb355023`, поэтому цепочка выглядит так:

```
dac6eb355023 (добавляла completed_at)  ->  a1b2c3d4e5f6 (title + tasktaglink)  [head]
```

`head` — это «самая свежая миграция в цепочке».

### `alembic/env.py` (изменили)

Добавили чтение адреса базы из переменной окружения `DATABASE_URL`:

```python
database_url = os.getenv("DATABASE_URL")
if database_url:
    config.set_main_option("sqlalchemy.url", database_url)
```

Зачем: в `alembic.ini` адрес базы захардкожен как локальный sqlite. Без этой
правки alembic на проде применял бы миграции не в Postgres, а в sqlite. Теперь:
- **на проде** есть `DATABASE_URL` (Postgres) → alembic работает с Postgres;
- **локально** переменной нет → остаётся sqlite из `alembic.ini`.

Так же адрес читает и само приложение (`database.py`) — теперь они согласованы.

### `prestart.py` (новый)

Небольшой скрипт, который **безопасно** приводит базу в актуальное состояние
перед стартом приложения. Он нужен, потому что база на проде могла оказаться в
одном из трёх состояний, и «в лоб» `alembic upgrade head` не во всех случаях
сработал бы (подробно — в разделе 4).

### `entrypoint.sh` (новый)

Скрипт запуска контейнера. Он:

1. вызывает `prestart.py` с повторными попытками (Postgres может стартовать
   чуть дольше бэкенда — до 15 попыток по 3 секунды);
2. если база так и не поднялась — останавливается с ошибкой;
3. если всё ок — запускает `uvicorn` (`exec` заменяет процесс, чтобы сигналы
   Docker корректно доходили до uvicorn).

### `Dockerfile` (изменили)

Раньше контейнер сразу запускал `uvicorn`. Теперь запускает `entrypoint.sh`,
который сначала готовит базу, а потом стартует приложение:

```dockerfile
RUN chmod +x /app/entrypoint.sh
CMD ["/app/entrypoint.sh"]
```

## 3. Что происходит при деплое (пошагово)

Workflow (`.github/workflows/deploy.yml`) мы **не меняли** — он по-прежнему делает
`docker compose up --build`. Всё новое происходит внутри старта контейнера бэкенда:

```
docker compose up --build
        │
        ▼
Стартует контейнер backend  ──►  CMD = entrypoint.sh
        │
        ├─ 1. python prestart.py   (готовит/мигрирует базу, с ретраями)
        │
        └─ 2. uvicorn main:app     (запускается приложение)
                     │
                     └─ lifespan → create_db_and_tables()
                        (теперь это просто «страховка»: всё уже создано,
                         поэтому фактически ничего не делает)
```

Ключевой момент: **`prestart.py` выполняется ДО того, как приложение вызовет
`create_all`**. Поэтому миграции применяются к «старой» схеме первыми, а
`create_all` в конце оказывается безобидным (создавать уже нечего).

---

## 4. Три сценария внутри `prestart.py`

Скрипт смотрит на текущее состояние базы и выбирает безопасный путь.

### Сценарий A — базой уже управляет Alembic

**Признак:** в базе есть таблица `alembic_version`.

**Действие:** обычный `alembic upgrade head` — докатываются только недостающие
миграции (в нашем случае `a1b2c3d4e5f6`).

### Сценарий B — таблицы есть, но истории Alembic нет

**Признак:** таблица `task` есть, а `alembic_version` — нет.
Это значит, что базу раньше строил только `create_all`.

**Проблема, которую решаем:** если бы мы просто запустили `upgrade head`, alembic
начал бы «с нуля» и попытался применить самую первую миграцию `dac6eb355023`
(она добавляет `completed_at`). Но эта колонка уже существует → ошибка.

**Действие:**
1. `alembic stamp dac6eb355023` — **помечаем** базу этой ревизией, ничего не меняя
   в схеме. Мы как бы говорим alembic: «состояние на этой ревизии у нас уже есть».
2. `alembic upgrade head` — теперь докатывается только новая миграция
   `a1b2c3d4e5f6` (title + tasktaglink).

> `stamp` — это «проставить версию без выполнения SQL миграции». Отличие от
> `upgrade` в том, что `upgrade` реально выполняет `upgrade()`-код, а `stamp`
> только пишет id в `alembic_version`.

### Сценарий C — пустая база (первый запуск)

**Признак:** нет ни `alembic_version`, ни `task`.

**Действие:**
1. `SQLModel.metadata.create_all(engine)` — создаём все таблицы сразу из моделей.
2. `alembic stamp head` — помечаем базу как «на самой свежей версии», чтобы в
   будущем alembic не пытался применять уже учтённые миграции.

## 5. Как добавлять новые изменения схемы в будущем

Когда снова понадобится изменить схему (новое поле, новая таблица и т.п.):

1. Поменяй модель (например, добавь поле в `tasks/models.py`).

2. Сгенерируй миграцию автоматически:
   ```bash
   cd backend
   alembic revision --autogenerate -m "краткое описание изменения"
   ```
   Alembic сравнит модели с базой и создаст файл в `alembic/versions/`.

3. **Обязательно прочитай** сгенерированный файл — autogenerate не идеален,
   иногда что-то нужно поправить руками (особенно с переименованиями и данными).

4. Проверь локально:
   ```bash
   alembic upgrade head      # применить
   alembic downgrade -1      # откатить на шаг назад (проверить downgrade)
   alembic upgrade head      # снова применить
   ```

5. Закоммить новый файл миграции вместе с изменением модели.

6. При деплое миграция применится автоматически (через `prestart.py`).

> Важно: `down_revision` у новой миграции должен указывать на текущий `head`.
> При autogenerate alembic проставляет это сам.

## 6. Полезные команды Alembic

```bash
cd backend

alembic current            # на какой версии сейчас база
alembic history            # вся цепочка миграций
alembic heads              # какая ревизия считается самой свежей
alembic upgrade head       # применить все недостающие миграции
alembic downgrade -1       # откатить одну миграцию назад
alembic stamp head         # проставить версию без выполнения SQL
```

Локально команды работают с sqlite (`database.db`), потому что `DATABASE_URL`
не задан. Чтобы выполнить их против Postgres, задай переменную окружения:

```bash
DATABASE_URL="postgresql://user:password@host:5432/hello_db" alembic current
```

## 7. Короткое резюме

- `create_all` умеет только создавать новые таблицы, но не менять существующие —
  поэтому для новых колонок нужен Alembic.
- Теперь при деплое `entrypoint.sh` → `prestart.py` применяет миграции
  автоматически, ещё до старта приложения.
- `prestart.py` защищён от трёх возможных состояний базы, так что деплой не
  сломается, даже если раньше alembic на проде не запускался.
- Для будущих изменений схемы: правишь модель → `alembic revision --autogenerate`
  → проверяешь → коммитишь. Остальное произойдёт само при деплое.
